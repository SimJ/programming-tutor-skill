# Onboarding — ヒアリングの具体手順

Phase 1 (Onboarding) の実施マニュアル。会話でまず聞くべきことと、曖昧な返答へのフォールバックをまとめる。

## Step 0: 言語の確定（Language selection）

教える言語（日本語 / English / 中文 / Español / Français）を最初に確定する。詳細は `SKILL.md` の「Language handling」節を参照。

判定手順:
1. ユーザーの **初回メッセージの自然言語** を見て、明らかならそれを採用（質問しない）。
2. 不明瞭なら AskUserQuestion で問う:
   - 質問文は英語フォールバック: "Which language would you like me to teach in? / 教える言語を選んでください"
   - 選択肢: `日本語` / `English` / `中文（简体）` / `Español` / `Français`
3. 以降、このドキュメント内の日本語表記は、実際のユーザー出力では選択された言語に翻訳して提示する。**このリファレンスファイル自体を翻訳する必要はない** — あくまで Claude が読む指示書だから。

言語切替リクエスト（「英語で教えて」「请用中文」「En español por favor」等）には即応し、1 回だけ確認コメントを添える。

## 聞くべき 4 項目

### 1. 対象技術 (What)

まずここ。全ての判断の起点。
- 候補が広いとき → AskUserQuestion で multi-select：「言語全般」「.NET/Spring Boot」「NestJS/React」「Rust」など
- 既に特定されているとき（「Reactをやりたい」と会話初手で言っている）→ 再質問しない、そのまま進む

### 2. 現在のレベル (Who)

以下の 4 段階を基準に AskUserQuestion：
- **完全な初心者**: プログラミング自体ほぼ初めて
- **他言語経験あり（初めての技術）**: 他の言語は書けるが、この言語/FW は初めて
- **基礎は書ける（応用を深めたい）**: 基礎構文は分かる。実践パターンを学びたい
- **中級以上（特定トピックを深掘り）**: 全体は書ける。特定の部分（非同期、型、テスト、アーキテクチャ等）に絞りたい

### 3. ゴール (Why)

目的によって教える内容の重心が変わる：
- **業務で使う** → 実務でよく使うパターン、落とし穴、テスト、チーム開発の慣習を重視
- **趣味プロジェクト** → 楽しく動くものを作るのを重視、細かい設計論は控えめ
- **資格/面接対策** → 頻出トピックを網羅、コーディングテスト形式の演習を追加
- **純粋な興味** → 言語の思想や設計背景を丁寧に、急がない

### 4. ペース (How much)

- 1 日あたり確保できる時間（15 分 / 30 分 / 1 時間 / 2 時間以上）
- 希望完了時期（特になし / 数週間 / 1〜2 ヶ月 / 3 ヶ月以上）

これで全体ボリュームと 1 ステップのサイズ感を決める。

## 環境確認 (Light check)

対象技術が決まったら、最低限の環境確認を行う。いきなり全部検証しようとせず、聞いて・実行させて・結果を見る順で。

### 最低限確認すべきこと

- OS（macOS / Windows / Linux / WSL）
- エディタ（VS Code が標準想定。IntelliJ、Rider、Vim、Cursor 等でも進められる）
- ランタイム/コンパイラのインストール状況

### 技術別の確認コマンド

| 技術 | 確認コマンド例 |
| --- | --- |
| Python | `python --version` or `python3 --version` |
| Node / NestJS / React | `node --version` / `npm --version` |
| TypeScript | `tsc --version`（なくても後で入れられる） |
| C# / .NET | `dotnet --version` |
| Java / Spring Boot | `java --version` / `mvn --version` or `gradle --version` |
| Rust | `rustc --version` / `cargo --version` |
| Go | `go version` |

学習者に上記を実行してもらい、結果を貼ってもらう。バージョンが古すぎる / 未インストールなら、**公式インストーラのリンクを 1 つだけ示す**。選択肢を広げすぎない。

`scripts/check_env.sh` に一括チェックスクリプトがある。bash を走らせる準備が学習者にあるならこれを勧める。

## VS Code 拡張の推奨

強く薦めるのは最初のセットアップ時の 1 回だけ。毎ターン拡張を勧めるのは禁止。

| 技術 | 最初に入れる拡張 |
| --- | --- |
| Python | Python (Microsoft) |
| TypeScript / Node | ESLint, Prettier |
| C# / .NET | C# Dev Kit |
| Java / Spring Boot | Extension Pack for Java, Spring Boot Extension Pack |
| React | ES7+ React snippets, ESLint |
| Rust | rust-analyzer |

## 曖昧な返答のフォールバック

よくある曖昧な返答と対処：

- **「何からやればいいか分からない」**: まず「何を作りたいか / 使いたいか」を聞く。作りたいものがなければ「Web アプリ寄り / データ分析寄り / 低レイヤー寄り」の 3 択にする。
- **「全部できるようになりたい」**: 全部は一度にできない。「最初の 1 ヶ月で何ができていたら嬉しいですか」に言い換える。
- **「簡単にお願いします」**: 何が難しいと感じるか聞く（英語? 数学? 概念?）。原因により教え方を変える。
- **レベル自己申告が怪しい**: 初回ステップは控えめから入り、2〜3 ステップで実力を見てから難易度調整することを伝える。

## 完了条件

Phase 1 を終えるには以下が揃っている必要がある：
- 対象技術が 1 つに絞られている（複数なら最初の 1 つを決める）
- レベル想定がある（外れていたら後で調整するが、まず仮置き）
- ゴールが言語化されている
- 環境が整っている（または整える手順を提示済み）

これらが揃ったら Phase 2（学習計画作成）へ移る。
