# Code Review — コードレビューの進め方

学習者が書いたコードをレビューするときの手順と観点。教育目的のレビューは業務レビューと重心が違う。

## レビューの 3 つの柱

1. **動くか** — まず動作確認。バグや型エラーを最優先で見つける。
2. **理解しているか** — 書けていても理由が分かっていないと応用が効かない。要所で「なぜこう書いた?」と聞く。
3. **次の一歩につながるか** — 今のレベルでの改善余地を 1〜2 点だけ指摘する。全部直そうとしない。

## 手順

### Step 1: 全体を読む (まずは黙読)

即座にコメントせず、まずコードを最後まで読む。概観がないと優先度がつけられない。

### Step 2: 可能なら動かす

- Bash tool が使えるなら実行する。
- コンパイラ/型チェッカがある言語（TypeScript, Rust, C#, Java など）は、先に通して errors/warnings をさらう。
- 依存関係やランタイムがなく動かせないときは、学習者に「手元で走らせた結果を貼って」と依頼する。

### Step 3: フィードバックを構造化して返す

以下の型で返す。順番重要。

```markdown
## レビュー結果 — <ファイル名 or ステップ名>

### 👍 良いところ
- （具体的に 1〜3 点。「◯◯がこういう理由で良い」）

### 🚨 直したい（バグ/壊れる）
- 指摘 1 (before/after スニペット、なぜ壊れるか 2〜3 文で)
- （なければこの節を省略してよい）

### ⚠️ 改善したい（設計/可読性）
- 指摘 1 (before/after、なぜ改善になるか)
- （多くて 2 つまで。多すぎると学習者が潰れる）

### 💡 好み・発展（任意）
- 参考までに、もっと慣れてきたらこう書くとよい、という話

### 次の一歩
- 「このまま次のステップへ」または「ここだけ直して再提出してください」
```

### Step 4: レベルに応じて深さを変える

- **初心者**: 直すべきは 1〜2 点まで。スタイルや命名の枝葉は触らない。「動いて読めれば今は合格」。
- **中級**: 設計・命名・責務分割まで踏み込む。ただし好み領域は💡にとどめる。
- **中級以上**: パフォーマンス、テスタビリティ、言語特有のイディオムまで。トレードオフを言語化する。

## よくあるバグ系観点 (言語共通)

- Off-by-one（ループ境界）
- Null / None / undefined のハンドリング漏れ
- 例外/エラーのスロー漏れまたは潰し過ぎ
- ミュータブルな共有状態
- 配列の破壊的変更を前提にしたコード
- 浮動小数点の等値比較
- 型変換の暗黙前提（`"1" == 1` など）

## よくある可読性/設計観点

- 関数が長い（30 行を超えたら分割を検討）
- ネストが深い（3 段以上は早期リターンを検討）
- 名前が意図を表していない（`data`, `tmp`, `x` が数箇所に散在）
- コメントがコードと一致していない／コメントがそもそも無い場面で必要
- マジックナンバー／マジックストリングの放置
- 責務が混ざっている（I/O とロジックが同一関数に）

## 教育的フィードバックの書き方

- 悪い: 「ここは `for i in range(len(arr))` ではなく `enumerate` を使いましょう」
- 良い: 「Python では `for i, x in enumerate(arr)` と書くのが慣用です。理由は (1) インデックスと要素を同時に取れるのでコードが短くなる、(2) `len` と `range` の組み合わせは Python らしくないとされることが多い、の 2 点です。この書き方は Step N で学んだ◯◯とも繋がります」

なぜそう書くのかの**言語の思想**や**他の場面での応用**まで触れられると、定着率が大きく上がる。

## 学習者が「完璧なコード」に拘ったとき

初心者が「もっときれいに書けるはず」「これじゃ業務で通用しないですよね」と自己批判に入ったら、次のように整理する：
- 「今のステップで求められる水準を満たしているかどうか」と「業務水準」は別物。
- 動く → 読める → きれい の順。今は 1〜2 段目で十分。
- きれいさは後から学ぶリファクタリングの章で扱う。

## レビューを断るとき

- コード量が 1 ファイル 500 行を超えるなど明らかに過大なら、「先にどの部分を見てほしいか」を確認する。全部を一度に見ると学習者も消化しきれない。
- コードに第三者の個人情報や機密が含まれているなら、レビューを止めてまず秘匿化してもらう。
